
## Setup

Before executing any script, install the requirements:

```bash
pip3 install -r requirements.txt
```

Create a `.env` file in the project root with your MongoDB connection string:
```bash
MONGO_URI=mongodb+srv://<user>:<password>@<cluster>.mongodb.net/spotify
```

### 1. Execute Python script to load the data

```bash
python3 scripts/01_load_data.py
```

> If there is a problem with certificates on Mac OS, run the following:
> ```bash
> /Applications/Python\ 3.14/Install\ Certificates.command
> ```

### 2. Execute JS script to transform tracks_raw into tracks

```bash
mongosh "MONGO_URI" --file scripts/02_transform.js
```

## Data Schema

The `spotify` database contains two collections: `tracks_raw` (loaded as-is from the source CSV) and `tracks` (produced by the transform script).

### tracks_raw

Flat structure mirroring the original CSV. Artists are stored as a single string, audio features are top-level fields, and there is no derived data.

```json
{
  "_id": { "$oid": "6a11c6b0ccf18b62c3e449cf" },
  "track_id": "5SuOikwiRyPMVoIQDJUgSV",
  "artists": "Gen Hoshino",
  "album_name": "Comedy",
  "track_name": "Comedy",
  "popularity": 73,
  "duration_ms": 230666,
  "explicit": false,
  "danceability": 0.676,
  "energy": 0.461,
  "key": 1,
  "loudness": -6.746,
  "mode": 0,
  "speechiness": 0.143,
  "acousticness": 0.0322,
  "instrumentalness": 0.00000101,
  "liveness": 0.358,
  "valence": 0.715,
  "tempo": 87.917,
  "time_signature": 4,
  "track_genre": "acoustic"
}
```

### tracks

Transformed structure with three key changes: `artists` is parsed into an array to support multi-artist tracks and array-based queries; audio features are grouped into a nested `audio_features` object; and two derived fields are added — `duration_sec` and `popularity_tier`.

```json
{
  "_id": { "$oid": "6a1ade9d93a7c1c61b0bb8c7" },
  "track_id": "5SuOikwiRyPMVoIQDJUgSV",
  "album_name": "Comedy",
  "track_name": "Comedy",
  "popularity": 73,
  "duration_ms": 230666,
  "explicit": false,
  "track_genre": "acoustic",
  "artists": ["Gen Hoshino"],
  "audio_features": {
    "danceability": 0.676,
    "energy": 0.461,
    "loudness": -6.746,
    "speechiness": 0.143,
    "acousticness": 0.0322,
    "instrumentalness": 0.00000101,
    "liveness": 0.358,
    "valence": 0.715,
    "tempo": 87.917,
    "key": 1,
    "mode": 0,
    "time_signature": 4
  },
  "duration_sec": 230.7,
  "popularity_tier": "high"
}
```

## Частина 1

### Питання

**1. Чому аудіо-характеристики винесені в окремий об'єкт `audio_features`, а не зберігаються плоско? Коли таке вкладення вигідне, а коли створює проблеми?**

Аудіо-характеристики винесені в окремий об'єкт `audio_features`, щоб логічно згрупувати пов'язані між собою дані та зробити схему більш структурованою і зрозумілою. Такий підхід відповідає документоорієнтованій моделі MongoDB, де вкладені об'єкти використовуються для представлення пов'язаних сутностей. Це також зменшує кількість полів на верхньому рівні документа та спрощує роботу з усім набором характеристик аудіо у вигляді одного блоку.

Проблемою може бути складніша робота із запитами та індексами через використання вкладених полів, особливо при глибокій вкладеності або частому доступі до окремих характеристик.

---

**2. Чому виконавці зберігаються як масив, а не як рядок? Які запити стають простішими?**

Виконавці зберігаються як масив, тому що масив є природним способом представлення таких даних. Це значно спрощує пошук треків за конкретним виконавцем, агрегацію статистики по артистах, також створення індексів. Зберігання у вигляді рядка вимагало б додаткового парсингу та використання regex, що менш ефективно і складніше в підтримці.

---

**3. Що таке `$out` і чим він відрізняється від `$merge`? Коли використовувати кожен?**

`$out` записує результат у нову або існуючу колекцію, повністю замінюючи її вміст. Він підходить для повної трансформації даних та побудови нової схеми.

`$merge` також записує результат у колекцію, але дозволяє оновлювати існуючі документи або додавати нові без повного перезапису колекції. `$merge` доцільно використовувати для інкрементального оновлення даних, синхронізації колекцій або сценаріїв де потрібно зберегти існуючі записи.

## Частина 2

### Питання

**1. Для чого використовується інструкція `$unwind`?**

`$unwind` використовується для деконструкції масивів. Цей оператор бере один документ, який містить масив із кількох елементів та перетворює його на кілька окремих документів-копій. У кожному новому документі на місці масиву буде стояти лише один його елемент. Це необхідно для того, щоб мати змогу фільтрувати, сортувати або групувати окремі елементи масиву за допомогою інших етапів агрегації. Приклад застосування можна побачити з полем `artists` у другому завданні.

---

**2. Чим `$stdDevPop` відрізняється від `$stdDevSamp`?**

Обидва рахують стандартне відхилення.

- **`$stdDevPop`** — розраховує відхилення для генеральної сукупності. Використовується коли набір даних є повним і остаточним; математично ділить на `N`.
- **`$stdDevSamp`** — розраховує відхилення для часткової вибірки. Використовується коли наявні дані — це лише частина від більшої кількості даних, і потрібно оцінити загальну тенденцію; математично ділить на `N−1`, що враховує похибку.

## Частина 3

### Питання

**1. У запиті 1 ми фільтруємо виконавців, у яких менше 5 треків. Як зміниться результат, якщо знизити поріг до 1? А що станеться, якщо вибирати виконавців із більш ніж 50 треками? Поясніть результат.**

Не менше 5 треків — це "хоча б 5 треків". Враховуючи що в нас колекція треків, то це по суті пройтися по всім трекам, оскільки швидше за все немає виконавців без хоча б одного трека. В такому випадку топ 10 повністю зміниться — туди потраплять виконавці одного хіта, який має високу популярність. Статистично це викривить результат, бо один трек не відображає стабільну популярність автора.

Якщо вибирати виконавців із більше аніж 50 треків, топ 10 також кардинально зміниться і звузиться. У вибірці залишаться лише продуктивні артисти з великими дискографіями. При цьому їхня середня популярність може впасти, бо прохідні пісні з альбомів тягнутимуть рейтинг вниз, а успішні артисти з кількістю треків < 50 відсіються.

---

**2. У запиті 3 ми фільтруємо жанри з менше ніж 100 треками. Чи зміниться результат, якщо знизити поріг до 50? Поясніть результат.**

Ні, результат не зміниться. З таким параметром повертає той самий жанр `"kids"`. Зниження ліміту до 50 лише допустить до конкурсу більш рідкісні жанри, але вони не перевершують популярну танцювальну музику за середніми показниками.

**1.2 Як зрозуміти, що індекс використовується? Наведіть скріншот або значення полів із `explain()`, які це підтверджують.**

Факт використання індексу можна визначити по полям `totalKeysExamined` та `totalDocsExamined`. Також можна подивитися на `inputStage`: якщо `COLLSCAN` — це пошук без індексу, `IXSCAN` — вказує на використання індексу.

**До індексу:**

```
executionSuccess: true,
nReturned: 354,
executionTimeMillis: 90,
totalKeysExamined: 0,
totalDocsExamined: 113999,
```

**Після індексу:**

```
executionSuccess: true,
nReturned: 354,
executionTimeMillis: 2,
totalKeysExamined: 354,
totalDocsExamined: 354,
```

**3.1 Чи є цей запит покривним (covered query)? Надайте розгорнуту та обґрунтовану відповідь.**

Ні, він не є покривним. Покривним називається запит, який може бути повністю виконаний лише за допомогою індексу без звернення до самих документів колекції.

У плані виконання є стадії `IXSCAN`, а потім `FETCH`. Наявність стадії `FETCH` означає, що MongoDB звертається до самих документів колекції для отримання додаткових даних. Також поле `"popularity"` не є проіндексованим. Але навіть якби воно було, в запиті немає проєкції, яка відокремлює індексовані поля, тому все одно для повернення запитів треба буде звернутися до документів. Через це це не є covered query.
